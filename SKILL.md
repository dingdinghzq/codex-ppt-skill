# Codex PPT

## Overview

This skill creates image-based PowerPoint decks from source material. Each slide is a complete 16:9 generated image. Final images are assembled into `.pptx` with `scripts/assemble_ppt.py`.

Use this when the user wants a visually unified presentation and accepts full-slide image pages. Do not use it when every textbox, chart, or shape must remain separately editable.

Prefer the built-in image generation/editing tool. Use `scripts/image_gen.py` only when the built-in backend is unavailable, lacks a required capability, or the user explicitly asks for API/CLI mode.

## Hard Constraints

- Read the relevant `Reference Map` files before each phase. This file is the orchestration contract; detailed rules live in `docs/` and worker prompts in `prompts/`.
- Use one upfront intake only. Present exactly five numbered clarification questions in one message before production starts: visual style, deck length, audience and purpose, content scope and required assets, and language and speaker notes.
- The visual-style question must present all 10 styles in `references/` with concise descriptions. Recommend one based on the topic, but let the user choose any listed style, provide a custom reference, or accept the recommendation.
- If the request already answers an intake item, show it as a captured choice or proposed default instead of asking an additional question later. Keep the intake to the same five numbered items.
- A normal initial request such as "create a deck" does not waive the intake. Present the five questions before production unless the user explicitly says not to ask questions or to use defaults immediately.
- Once the user answers the intake or explicitly says to proceed, continue through outline, style, backend, style anchor, slide generation, QA, notes, and assembly without asking more questions.
- If the user says to use judgment, accepts the defaults, or explicitly says to create/proceed without answering every item, lock the recommended defaults and begin production.
- Do not ask the user about backend selection, style-anchor approval, output-path confirmation, or parallel-worker authorization. Infer those operational choices from available tools and context. When workers are unavailable or not already authorized by the runtime, use parent-local generation and keep going.
- The main agent owns orchestration, prompt jobs, state recording, QA, speaker notes, and assembly.
- Every final `origin_image/slide_XX.png` must be generated by the selected image backend: built-in image generation/editing tool or `scripts/image_gen.py`.
- Local drawing, Pillow, SVG, HTML/CSS/canvas screenshots, python-pptx/PptxGenJS layouts, and manual overlays are failure modes, not fallbacks.
- Fix the selected image backend before generating the style anchor. Do not let workers switch backend for convenience.
- Generate one representative style-anchor slide, QA it internally, record how it was generated, and pass that exact method to every remaining slide job. Do not ask the user to approve the anchor.
- Slide dispatch and result state must be recorded with the bundled scripts. Chat messages alone do not make a slide dispatched or complete.
- If a required backend or required-image path is unavailable, retry reasonable transient failures, then report a blocker with the slide id and evidence. Do not ask a new mid-production question or create a lower-quality replacement.

## Visible Progress

For non-trivial decks, keep a user-visible checklist with one active step. Canonical completion evidence is in `docs/workflow-gates-and-progress.md`.

Default visible steps:

1. Complete upfront intake and prepare source decisions.
2. Generate and internally accept one style-anchor slide.
3. Prepare slide jobs and slide state.
4. Dispatch available slide workers or parent-local jobs.
5. Record generated slide results.
6. QA, repair, notes, and PPT assembly.

Do not mark a step complete from chat alone; use real files or script-recorded state.

## Default Workflow

1. Understand the source content.
   - Identify topic, audience, goal, page count, style/brand constraints, and sections to include or exclude.
   - Prepare recommendations for any missing intake choices before asking the five questions.

2. Run the upfront intake.
   - Before asking anything, inspect the request, supplied files, and available tools.
   - Read `docs/workflow-gates-and-progress.md` and `docs/outline-style-and-sample.md`.
   - Ask exactly five numbered questions in one message:
     1. Visual style: list all 10 built-in reference styles with short descriptions and mark one recommendation.
     2. Deck length: ask for an exact slide count or offer short, standard, and detailed ranges.
     3. Audience and purpose: ask who will view the deck, what they should learn or decide, and the presentation setting.
     4. Content scope and required assets: ask for must-include points, exclusions, source files, logos, charts, or screenshots.
     5. Language and speaker notes: ask for presentation language and whether notes should be included.
   - Prefill choices already known from the request. Recommend defaults for unresolved items in the same message.
   - Invite one reply covering all five items, or allow "use the defaults."
   - Production begins after the user replies to the intake, accepts defaults, or says to use judgment/proceed. The initial deck request alone is not a reply to the intake. Do not pause for further confirmations or checkpoints.

3. Plan the deck outline.
   - Before writing or updating `outline.md`, read `docs/workflow-gates-and-progress.md` and `docs/outline-style-and-sample.md`.
   - Write the final outline using the intake decisions and inferred defaults. Do not stop for outline approval.

4. Choose a unified visual style.
   - Before including style options in upfront intake or using files from `references/`, read `docs/outline-style-and-sample.md`.
   - Use the intake choice or supplied reference. If the user accepts defaults or does not select a style before explicitly proceeding, use the recommended reference style and record the decision without asking again.

5. Select the image backend.
   - Before generating any slide image, read `docs/backend-selection.md`.
   - Check whether a built-in image tool is callable and select the backend using the documented rules. Report the choice as a progress update, not a question.
   - If CLI/API fallback is selected, read `docs/cli-api-fallback.md`. Read `docs/image-model-configuration.md` only after config errors or explicit API-setting requests.

6. Generate one style-anchor slide.
   - Before generating the style anchor, read `docs/outline-style-and-sample.md`.
   - Generate exactly one representative content slide, inspect it internally, and repair it until it is a suitable deck-wide reference.
   - Record `sample_generation_method` in `deck_spec.json` and mark the anchor slide accepted in run state. The legacy field name remains for script compatibility.
   - Continue directly to full-deck production without asking for approval.

7. Create the project directory.
   - Before initializing folders or assembling files, read `docs/project-assembly-and-reporting.md`.
   - If no destination is specified, use the current working directory or the source file directory.

8. Prepare user-supplied assets.
   - Before using paper figures, charts, screenshots, logos, or other required assets, read `docs/user-supplied-assets.md`.
   - Treat required assets as strict inputs. Resolve slide-to-asset mapping during the upfront intake; after production starts, use that mapping and do not ask again.

9. Generate all slide images.
   - Before full-deck image generation, read `docs/slide-generation-and-subagents.md`.
   - Create per-slide jobs with `scripts/prepare_slide_prompts.py` or saved `prompts/slide_XX.json` files.
   - Every final image must come from the selected backend and be recorded with bundled state scripts.

10. Dispatch slide jobs.
   - Before dispatching or replacing slide workers, read `docs/slide-generation-and-subagents.md` and `prompts/slide-worker.md`.
   - When the runtime already permits spawning workers, use one worker per remaining slide job whenever possible.
   - Otherwise generate jobs parent-locally, recording dispatch with a stable local agent id such as `parent-local`. Do not request authorization during intake or production.

11. Quality check and repair.
    - Before QA or assembly, read `docs/project-assembly-and-reporting.md`.
    - Inspect every slide before assembly: text, outline match, truncation, style, unwanted page numbers, overlaps, and required assets.
    - Regenerate severe failures with a tighter prompt. Use backend editing for localized issues when available.
    - For CLI/API fallback edit commands, read `docs/cli-api-fallback.md`. Replace the final slide only after validating the edited output.

12. Write speaker notes and assemble the PPT.
    - Before writing `speech.md` or running assembly, read `docs/project-assembly-and-reporting.md`.
    - Make sure `outline.md` reflects the final deck outline. Use `speech.md` headings that map to `Slide N`.
    - Before assembly, ensure `slide_jobs.json` shows generated slides as `recorded` and the style anchor as `accepted`. If any slide is `pending`, `dispatched`, or `blocked`, stop and report the state without asking a new question.

13. Report the result.
    - Use the final report checklist in `docs/project-assembly-and-reporting.md`.
    - Include paths, slide count, backend used, recorded-result status, and any limitations or blockers.

14. Save reusable styles when requested.
    - If asked to save the current deck style or a supplied image/PDF/PPT/PPTX style, read `docs/style-library.md`.

## Slide Worker Dispatch

Use slide subagents after the style anchor when the runtime already permits them. Otherwise use parent-local generation without interrupting the workflow. The main agent prepares jobs and records state; each worker handles exactly one `prompts/slide_XX.json` job and returns only selected image path, backend, and QA note.

Use `docs/slide-generation-and-subagents.md` for dispatch, commands, result recording, blockers, and backend provenance. Use `prompts/slide-worker.md` as the handoff template.

Subagents must not edit `outline.md`, `deck_spec.json`, other slide jobs, `origin_image/`, `speech.md`, or the final `.pptx`. The parent records outputs and assembles.

## Acceptance Criteria

- Output is a valid `.pptx`.
- Each expected final slide image exists under `origin_image/slide_XX.png`.
- Every final slide image was generated by the selected backend and recorded through `record_slide_result.py`, except the internally accepted style anchor marked accepted by run state.
- `outline.md` reflects the final deck outline.
- `speech.md` exists when speaker notes are expected, and assembly writes those notes into the PPT.
- `slide_jobs.json` and `slide_run_state.json` reflect the final state.
- Required source images are visibly represented, or a blocker is reported.
- If blocked, the final response identifies phase, slide id, evidence path, and unfinished reason; do not call the deck complete.
